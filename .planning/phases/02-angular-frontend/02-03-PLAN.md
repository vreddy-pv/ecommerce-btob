<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Cart submission → POST /api/orders | Order creation crosses to backend; accountId in body (not JWT-extracted) |
| Order history response → mat-table | OrderResponse data rendered in DOM |
| Route param :id → GET /api/orders/{id} | User-controlled path segment reaches backend |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-10 | Tampering | CartSidebarComponent order submission | mitigate | accountId read from authService.account() (not user input); items mapped from cartService (not user-controlled raw); backend validates CreateOrderRequest (Jakarta Validation Phase 1) |
| T-02-11 | Spoofing | OrdersComponent getOrders accountId | mitigate | accountId sourced from authService.account() signal (populated from JWT AuthResponse, not URL/user input); user cannot view another account's orders because backend filters by accountId param |
| T-02-12 | Information Disclosure | OrderDetailComponent :id route param | accept | User can view their own order by ID; backend does not currently enforce ownership (Phase 1 gap noted in RESEARCH Pitfall 2 — catalog/order endpoints have no JWT validation), but this is a backend concern not a frontend mitigation; frontend only displays what backend returns |
| T-02-13 | Tampering | StatusChipComponent status display | mitigate | Status rendered from OrderResponse.status enum (backend-controlled); chip color is display-only — no status mutation from frontend (ORDER-02 is display, not update, per RESEARCH Open Question 3) |
| T-02-14 | Elevation | order creation with another account's accountId | accept | Frontend reads accountId from authenticated session; a malicious user could craft a POST with another accountId, but backend order-service has no JWT validation (Phase 1 gap) — this is a known backend limitation, not a frontend regression; documented in RESEARCH Pitfall 2 |
</threat_model>
